## skill-creator
创建一个会议skill，它拥有会议秘书的专项技能，根据会议纪要文件或者录音，生成会议纪要，将重复的，繁琐的工作自动化，向OPC迈进

固定的结构和模式
- 文件夹
 - SKILL.md
 重复的工作，专业的技能，skill.md会记录下来，将prompt固化在我们的项目之中
 - skill-creator
 anthorpic官方推出的 skill creator，它可以帮助我们快速创建一个标准化，简化的skill封装
 会有自己封装的skill的需求，当我们频繁重复的时候

- SKILL.md 固定格式
1. 
文件开头用 --- 包裹的 YAML 块，必填字段有两个：
---
name: skill-name           # 技能标识符（英文，kebab-case）
description: 技能描述       # 何时触发、做什么（这是触发匹配的关键）
---

name：技能的唯一标识名，用英文短横线连接，如 meeting-minutes、skill-creator
description：这是 最重要的字段。Claude 根据这段描述来决定是否触发该技能。不要只写功能，还要写明什么场景/关键词下使用。 
2. Markdown 正文（指令体）
YAML 块之后是 Markdown 格式的指令正文，没有强制结构，但推荐包含以下模块：
- 处理流程：分步骤说明技能如何完成任务
  - 例：第一步通读全文 → 第二步清洗内容 → 第三步结构化输出
- 输出模板：用代码块定义产出的固定格式（让 AI 严格照模板填）
- 质量要求：声明输出标准（完整性、准确性、简洁性、诚实性）
- 示例：Input → Output 对照，一个完整例子胜过千言万语
- 注意事项：边界情况怎么处理（信息缺失留空不编造、多人同时发言标[听不清]等）

写作原则：
- 用祈使句（"通读全文"、"按模板输出"），不要用商量语气
- 解释为什么这么做，而不是堆砌 ALWAYS / NEVER
- 字数控制：SKILL.md 正文尽量 500 行以内，超出就拆到 references/ 目录
- 示例驱动：好的 Input/Output 例子比纯文字描述有效得多
